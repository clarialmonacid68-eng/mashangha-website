# 码上好正式运营启动清单

## 当前结论

网站现在适合作为“试运营获客页”，但还不能作为完整交易平台运营。

已具备：

- 自定义域名与 Vercel 静态部署
- 首页、需求展示、开发者展示、需求登记、开发者入驻页面
- 桌面端和移动端基础响应式布局
- 搜索引擎描述、`robots.txt`、`sitemap.xml` 和 404 页面
- 试运营与示例数据标识

尚未具备：

- 真实表单收件和后台线索管理
- 登录、注册和用户身份系统
- 真实需求、开发者和项目数据库
- 消息、报价、订单和交付流程
- 支付、退款、发票或资金托管能力
- 正式服务协议、隐私政策和运营主体信息
- 数据分析、错误监控和转化统计

## 上线前必须完成

### P0：先让线索能够被收到

1. 确定需求提交后的收件渠道：
   - 企业微信或个人微信
   - 企业邮箱
   - 飞书多维表格、腾讯文档或独立数据库
2. 确定开发者入驻申请的收件渠道。
3. 给两个表单接入真实提交接口，并增加：
   - 必填校验
   - 联系方式格式校验
   - 用户同意隐私说明
   - 成功和失败反馈
   - 防垃圾提交
4. 每天至少检查两次新线索，并建立跟进状态：
   - 新提交
   - 已联系
   - 需求已确认
   - 匹配中
   - 已成交
   - 暂不跟进

在 P0 完成前，不建议购买广告流量。

### P1：建立可信的首批供给

1. 人工招募 5-10 位真实开发者。
2. 核验身份、作品链接、技能、可接单状态和价格范围。
3. 获得本人同意后再公开展示资料。
4. 删除所有示例人物，换成真实且可核验的数据。
5. 先采用人工匹配，不急着开发复杂的自动报价系统。

### P1：获得首批真实需求

1. 选择一个清晰的早期客群，例如：
   - 本地商家小程序
   - 企业内部管理工具
   - AI 客服与知识库
   - 自动化工作流
2. 为该客群准备 3 个标准服务包：
   - 交付内容
   - 大致价格区间
   - 预计周期
   - 不包含的范围
3. 通过熟人、行业微信群、内容平台和线下商家获得前 10 条需求。
4. 每条需求都记录来源、预算、成交原因和流失原因。

### P1：补齐信任与合规页面

正式收集联系方式之前，至少准备：

- 运营主体或负责人信息
- 联系方式
- 隐私说明
- 服务规则
- 投诉和纠纷处理方式
- 数据删除或撤回渠道

在没有取得相应能力前，不要宣传“平台资金托管”“担保交易”“保证退款”等服务。

## 推荐的最小技术方案

第一阶段无需立刻重构 Next.js，可以保留静态页面：

- 前端：现有 HTML/CSS
- 表单：Vercel Serverless Function
- 数据库：Supabase PostgreSQL
- 通知：企业邮箱或企业微信机器人
- 统计：Vercel Web Analytics 或其他合规统计工具
- 线索后台：Supabase Dashboard，后续再开发管理后台

建议先建两张表：

### `demand_leads`

- `id`
- `title`
- `project_type`
- `description`
- `budget`
- `delivery_time`
- `cooperation_mode`
- `contact_name`
- `contact_value`
- `status`
- `source`
- `created_at`

### `developer_applications`

- `id`
- `name`
- `contact_value`
- `city`
- `specialty`
- `tools`
- `starting_price`
- `portfolio_url`
- `status`
- `created_at`

## 30 天运营节奏

### 第 1 周

- 接通两个表单
- 补齐隐私说明和联系方式
- 招募 5 位真实开发者
- 确定一个主要客户群

### 第 2 周

- 获取 10 条真实需求
- 完成人工需求访谈
- 建立开发者匹配表和跟进表

### 第 3 周

- 推进 2-3 个真实报价
- 记录常见需求、预算异议和流失原因
- 根据真实反馈调整首页文案

### 第 4 周

- 争取完成首个付费项目
- 取得可公开的案例与评价授权
- 决定是否投入用户系统、订单系统和支付能力

## 需要运营者提供的资料

要继续完成 P0，需明确：

1. 需求线索发送到哪里。
2. 开发者申请发送到哪里。
3. 网站公开显示的联系人、微信或邮箱。
4. 运营主体名称；若暂未注册公司，明确当前使用个人还是筹备中主体。
5. 是否已有 Supabase、企业邮箱或企业微信。

## 第一阶段平台运营手册

当前系统已进入“本地可验证的第一阶段交易平台”状态：支持账号、需求、开发者审核、报价、模拟全额付款、订单协作、交付验收、评价、争议和站内通知。真实支付、真实退款、分账和对账仍未上线。

### 每日审核

1. 每天至少两次进入 `/admin/developers` 和 `/admin/demands`。
2. 开发者审核必须核验资料完整性、作品链接、联系信息、技能范围和收款主体信息。
3. 需求审核必须确认描述清楚、预算合理、无违法违规内容、无绕开平台交易的联系方式。
4. 所有通过或拒绝动作必须填写审核备注；系统会写入审计日志。
5. 被拒绝的资料需要给出可执行原因，例如“作品链接无法访问”或“预算与范围不匹配”。

### 异常订单

1. 每天查看 `/admin/orders`，重点关注长时间停留在 `pending_payment`、`delivered`、`disputed`、`refund_review` 的订单。
2. 发现异常支付、重复支付、关闭支付单或失败支付时，在 `/admin/risk` 标记并人工核对。
3. 冻结或暂停后续操作时，只阻止新动作，不修改历史财务记录。
4. 对高风险订单，保留订单状态历史、留言、附件路径、交付版本和支付记录截图。

### 投诉与争议

1. 争议入口为订单争议流程，运营人员在 `/admin/disputes` 查看。
2. 处理争议前，必须收集双方陈述、交付版本、附件证据、聊天记录和需求/报价原文。
3. 第一阶段裁决只允许三类结论：继续履约、验收结算、全额退款。
4. 裁决说明必须写清依据，并写入审计日志。

### 退款处理

1. 未支付订单不可退款。
2. 第一阶段只支持未分账前的全额退款策略模型。
3. 已完成分账的订单不能走普通未分账退款路径，必须进入人工财务处理。
4. 退款渠道失败时，退款记录标记为失败，订单回到 `refund_review`。
5. 只有管理员确认失败原因后，才允许再次触发退款。

### 证据保全

1. 订单留言、附件路径、正式交付版本、验收/拒绝理由、争议原因和裁决说明都属于证据。
2. 不得删除争议订单相关数据；确需隐藏违规内容时，只做隐藏标记并保留审计记录。
3. 外部沟通材料应上传或登记到对应争议记录，避免只存在个人聊天工具中。

### 数据删除

1. 用户请求删除个人资料时，先确认是否存在未完结订单、争议、退款或审计义务。
2. 可删除或匿名化展示资料、联系方式和营销信息。
3. 订单、支付、退款、争议和审计日志在法务/财务留存期内不得硬删除。
4. 删除或匿名化动作必须记录处理人、时间、原因和影响范围。

### 支付能力宣传限制

在真实支付商户、退款、分账、对账和生产证书全部验收前，页面和销售话术不得宣传：

- 平台资金托管
- 担保交易
- 自动分账
- 保证退款
- 支付即受监管资金保护

当前只能表述为“本地模拟全额付款流程”或“第一阶段交易流程验证中”。对外成交建议继续使用线下合同和人工收款，直到微信支付资金闭环完成。
