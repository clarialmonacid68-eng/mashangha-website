## 4. 主要问题与风险

### P1-1：默认全量 Playwright 门禁失败，认证 E2E 存在稳定性风险

**证据**

- `playwright.config.ts:5` 设置 `fullyParallel: true`。
- 默认全量运行 17 项时，`tests/e2e/auth.spec.ts:124` 的手机号 OTP 登录未跳转到 `/workspace/settings`，最终停留在 `/verify?phone=...`；16 项通过、1 项失败、进程 exit 1。
- 该测试使用固定手机号 `+8613800138000`，并在开始时删除匹配手机号及所有 `e2e-` 前缀用户（`tests/e2e/auth.spec.ts:16-44`），存在共享认证环境和清理范围耦合。
- 单项、单 worker 复跑通过，说明并非稳定功能缺失，但也说明默认并行门禁的可重复性不足。

**影响**

默认交付命令不能稳定给出绿色结果。认证是所有业务链路的入口，不能把单项复跑通过当成全量通过。

**建议**

1. 先通过重复运行、trace、浏览器 console/network 与 Supabase Auth 日志确认失败发生在点击事件、OTP 校验、会话写入还是跳转阶段。
2. 缩小测试清理范围，使用每次测试唯一的手机号或将固定 OTP 测试串行化。
3. 在 CI 中执行默认并行配置并至少连续通过多次，再关闭此问题。

### P1-2：生产迁移通过 SQL Editor 手工执行，缺少可审计的迁移历史

**证据**

- `docs/deployment/tencent-cloud-preflight.md:151` 声明生产库已在 SQL Editor 执行聚合文件 `docs/deployment/generated/production-migrations.sql`。
- 聚合文件包含 17 个迁移正文，但没有向 Supabase CLI 的迁移历史登记对应版本。
- 当前证据只确认 Table Editor 能看到部分表，未证明全部函数、触发器、RLS policy、grant、bucket 和迁移版本与仓库一致。

**影响**

后续使用 `supabase db push` 时可能把已执行迁移识别为未执行并重复应用；生产 schema 漂移也无法可靠审计或自动化恢复。

**建议**

1. 导出生产 schema，与本地 reset 后 schema 做结构化 diff。
2. 使用受支持的 linked migration 流程部署；如保留现有库，应在核验后修复迁移历史，而不是继续手工拼接 SQL。
3. 把生产迁移、备份和回滚步骤纳入发布流水线，并保留执行记录。

**建议**

### P2-1：部署交付物仍未进入版本控制，工作区边界不完整

### P2-2：`commit-session.sh` 已过期，不可作为当前安全提交工具

**证据**

- 脚本从 `commit-session.sh:51` 开始再次 stage 并提交已存在于 HEAD 历史中的迁移、UI 和测试文件。
- 当前这些路径没有对应未提交改动，实际运行会在第一个 `git commit` 因“nothing to commit”退出。
- 脚本注释声称最终只剩 `.claude/` 和 `__pycache__/`，与当前工作区不符。

**影响**

工程师若按文档直接运行，会中途失败；更糟的是未来若这些文件有无关改动，脚本会按过期分组误提交。

**建议**

删除该一次性脚本，或重写为只检查/打印当前部署材料，不自动按历史路径批量提交。

### P2-3：限流和通用幂等仍是单进程内存实现

**证据**

- `lib/security/rate-limit.ts:11-12` 使用进程内 `Map`。
- `lib/security/idempotency.ts:1-2` 使用进程内 `Map`。

**影响**

PM2 多实例、进程重启或未来横向扩容时无法提供全局限流和幂等保证。支付 RPC 有数据库唯一约束是积极措施，但通用写接口仍不能依赖单进程状态作为生产安全边界。

**建议**

生产前迁移到数据库/Redis/KV 等共享存储，并为所有资金和订单状态变更设置数据库级唯一约束或事务状态门禁。

### P2-4：生产运维门禁尚不完整

当前没有完整证据证明以下事项已落地：HTTPS 强制跳转、生产 Auth 全流程、SMTP/短信、数据库自动备份与恢复演练、日志集中化、告警、健康检查、发布回滚、秘密轮换、三网访问测试、性能/容量测试、文件生命周期策略、运营客服 SOP。

这不是第一阶段本地闭环的否定，但它们是正式试运营前的必要条件。

## 7. 建议处理顺序

1. **先恢复测试可信度**：定位并修复默认并行 E2E 的 OTP 登录不稳定，连续运行全量套件通过。
2. **修正部署事实源**：更新 README 和腾讯云 runbook，明确 IP 已部署、域名未解析、认证/HTTPS未验收。
3. **规范生产数据库**：做 schema diff，补齐受控迁移历史、备份和回滚证据。
4. **整理并提交部署交付物**：只提交经核验材料，清理 `__pycache__` 和过期 `commit-session.sh`。
5. **完成预生产验收**：从生产认证到订单、文件、后台治理跑一轮真实环境 smoke test，并接监控告警。
6. **备案、域名、HTTPS和通信服务完成后再开放试运营**。
7. **微信支付准入完成后另立阶段接入真实支付、退款、分账和对账**，不得在当前 mock 实现上直接对外收款。
