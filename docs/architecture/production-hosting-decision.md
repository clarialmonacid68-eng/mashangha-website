# 生产部署决策

**状态：** 临时决策，待中国大陆网络实测  
**日期：** 2026-06-15

## 当前决策

第一阶段开发保持标准 Next.js + PostgreSQL 接口边界，不把业务逻辑绑定到 Vercel 专有服务。

当前静态原型已部署在 `https://www.mshcode.com/`。2026-06-15 检查确认：

- 返回 HTTP 200。
- 响应服务器为 Vercel。
- `x-vercel-id` 显示新加坡 `sin1` 区域。
- 当前网络 10 次采样 TTFB 约为 0.288–0.422 秒。

开发环境暂定：

- Next.js 本地运行
- Supabase 本地 CLI 提供 PostgreSQL、Auth 和 Storage
- 模拟支付适配器

生产环境暂不锁定 Vercel + Supabase。最终选择由 24 小时大陆网络实测和备案路径共同决定。

## 候选方案

### A. Vercel + Supabase

优点：

- Next.js 部署简单。
- 开发速度快。
- Auth、PostgreSQL 和 Storage 集成成本低。

风险：

- 官方区域配置不等于中国大陆网络质量保证。
- Supabase 托管手机号短信商不直接覆盖国内短信申请流程。
- 使用中国大陆服务器备案时，不能直接沿用该部署组合。

### B. 香港或新加坡 Node 托管 + PostgreSQL

优点：

- 不依赖中国大陆服务器备案后才能启动开发测试。
- 可选择更靠近大陆用户的区域。
- Next.js、PostgreSQL 和 S3 兼容存储迁移成本可控。

风险：

- 大陆跨境网络仍需实测。
- 运维、备份和安全责任高于全托管方案。

### C. 中国大陆云

优点：

- 通常更适合大陆用户访问。
- 国内短信、对象存储、监控和备案流程可由同一云厂商支持。

风险：

- 必须先完成域名实名认证和备案。
- 迁移到特定云产品会增加供应商耦合。

## 强制架构边界

- 数据访问集中在 `lib/db/`，页面不直接依赖 Supabase SDK。
- 认证集中在 `lib/auth/`，业务表使用内部用户 ID，不依赖短信供应商字段。
- 文件访问集中在 `lib/storage/`，只暴露上传、签名下载和删除接口。
- 定时任务通过普通受保护 HTTP handler 实现，不依赖单一平台调度语义。
- 支付通过 `PaymentProvider` 接口隔离。

## 生产决策所需输入

- `mshcode.com` 的注册主体和当前备案状态。
- 可使用的中国大陆云账号和备案接入商。
- Vercel、香港/新加坡节点的 24 小时大陆网络测试结果。
- 数据存储地区与隐私政策要求。

## 结论

可以开始渠道无关的本地开发；不得在完成实测前把 Vercel + Supabase 写成生产承诺。
