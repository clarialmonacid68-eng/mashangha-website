# 码上好 AI 开发交易平台设计规格

**日期：** 2026-06-15  
**状态：** 已确认  
**目标周期：** 1–2 个月  
**开发资源：** 1 名全栈开发者  
**首发终端：** 响应式 Web

## 1. 产品目标

将现有静态展示网站升级为面向客户与 AI 开发者的双边交易平台。平台覆盖需求发布、开发者认证、报价选择、全额付款、项目履约、交付验收、微信支付分账、评价和争议处理。

首版不追求自动匹配算法、实时聊天或复杂付款计划。平台先通过清晰、可审计的交易流程完成真实订单闭环。

## 2. 已确认的产品决策

- 同一账号可以同时作为客户和开发者，并在工作台切换角色。
- 登录支持手机号验证码和邮箱验证码。
- 视觉方向采用“专业可信 + 品牌活力”：营销页面保留暖色和亲和感，工作台采用清晰克制的 SaaS 风格。
- 首版仅支持单笔全额付款，不支持定金尾款或里程碑分期。
- 每个主订单只有一次正式交付、一次验收和一次分账。
- 已付款后的新增范围通过补充订单处理，不直接改写原订单金额。
- 沟通采用订单内留言与文件上传，不做实时聊天和音视频。
- 争议由平台人工仲裁，必要时邀请第三方专家评审。
- 平台仅向开发者收取成交佣金，客户不另付平台服务费。
- 支付渠道优先接入微信支付的平台型产品、二级商户和分账能力。
- 平台不建设用户余额、充值或自行提现功能。

## 3. 建设路径

采用两期交付。

### 第一期：业务闭环

预计 4–6 周，完成：

- Next.js 工程与设计系统
- 手机号和邮箱验证码登录
- 统一账号与双角色切换
- 开发者入驻、认证和资料展示
- 需求发布、审核、列表和详情
- 开发者报价、客户选标
- 订单生成、订单留言、附件和正式交付
- 客户验收、评价和仲裁申请
- 站内通知
- 用户、审核、需求、报价、订单和仲裁后台
- 模拟支付状态，用于支付资格审批期间联调和业务验收

### 第二期：真实资金闭环

预计 2–4 周，并与微信支付产品申请并行推进：

- 开发者二级商户入驻
- 微信支付下单和支付结果查询
- 支付回调验签与幂等处理
- 验收后的订单分账
- 退款申请、退款执行和退款查询
- 分账失败重试与异常告警
- 支付账单、平台订单与分账结果对账
- 后台支付、退款和结算管理

若支付产品资格延期，第一期仍可用于招募、审核和人工撮合，但不得宣传已提供资金托管或担保交易。

## 4. 信息架构

### 4.1 公开网站

- 首页：平台价值、服务流程、服务范围、真实案例、双端行动入口
- 需求市场：需求列表、筛选、排序和需求详情
- 开发者市场：开发者列表、筛选和开发者详情
- 发布需求：项目描述、预算、周期、附件和联系方式
- 开发者入驻：身份资料、技能、作品、服务范围和收款主体资料
- 规则与帮助：服务协议、隐私政策、交易规则、退款规则、争议规则和联系方式

未登录用户可以浏览公开内容。发布需求、报价、付款和订单操作必须登录。

### 4.2 客户工作台

- 工作台概览
- 我的需求
- 收到的报价
- 我的订单
- 订单留言与附件
- 交付验收
- 退款与仲裁
- 评价
- 通知中心
- 账号与安全设置

客户核心路径：

`发布需求 → 平台审核 → 接收报价 → 选择开发者 → 全额付款 → 查看履约记录 → 接收正式交付 → 验收或申请仲裁 → 评价`

### 4.3 开发者工作台

- 工作台概览
- 开发者认证资料
- 可报价需求
- 我的报价
- 履约订单
- 订单留言与附件
- 正式交付
- 收入与结算
- 通知中心
- 账号与安全设置

开发者核心路径：

`完成认证 → 浏览需求 → 提交报价 → 客户选标 → 等待付款 → 开始履约 → 提交正式交付 → 等待验收 → 分账结算`

### 4.4 运营后台

- 数据概览
- 用户管理
- 开发者认证审核
- 需求审核
- 报价管理
- 订单管理
- 支付查询与对账
- 退款管理
- 分账管理
- 仲裁与专家评审
- 风控事件
- 内容和费率配置
- 管理员与操作日志

## 5. 技术架构

### 5.1 应用层

- Next.js
- TypeScript
- Tailwind CSS
- 营销页面、用户工作台和运营后台使用同一代码库
- 使用独立路由组和布局区分公开区、工作台和后台

现有 HTML/CSS 作为内容与视觉参考，不继续在静态文件中叠加业务逻辑。重构时逐页迁移，并复用已有品牌色、文案和页面信息。

### 5.2 数据与认证

- Supabase PostgreSQL
- Supabase Auth
- 手机号验证码登录
- 邮箱验证码登录
- 账号支持同时拥有客户和开发者角色
- 使用数据库约束、服务端授权和行级安全策略共同隔离数据

手机号验证码依赖短信服务和模板审核。若 Supabase 在目标地区的手机号发送能力不满足要求，认证层需要接入国内短信服务，但账号模型保持不变。

### 5.3 文件存储

- Supabase Storage
- 公开文件：已审核的开发者头像和作品展示图
- 私有文件：需求附件、订单附件、正式交付物和仲裁证据
- 私有文件通过短时签名链接访问
- 服务端校验文件归属、扩展名、MIME 类型和大小

### 5.4 支付适配层

支付模块独立封装，业务订单不保存微信专属状态作为唯一事实来源。

支付适配层负责：

- 创建支付单
- 查询支付状态
- 验证支付回调
- 发起退款
- 查询退款状态
- 创建分账
- 查询分账结果
- 下载和解析账单

本地和测试环境使用模拟支付适配器；生产环境使用微信支付适配器。未来增加支付宝时，不修改订单核心状态机。

### 5.5 部署与运维

- Vercel 部署 Next.js
- Supabase 托管 PostgreSQL 和 Storage
- 定时任务处理待查单、超时提醒、分账重试和日常对账
- 错误监控覆盖前端异常、服务端异常和支付回调失败
- 生产环境密钥只保存在服务端环境变量中

## 6. 核心数据模型

### 6.1 身份与资料

- `profiles`：账号基础资料
- `user_roles`：客户、开发者和管理员角色
- `developer_profiles`：开发者公开资料和认证状态
- `developer_skills`：技能标签
- `portfolio_items`：开发者作品
- `developer_payment_accounts`：二级商户申请和审核状态

### 6.2 需求与报价

- `demands`：需求标题、描述、预算、周期和审核状态
- `demand_attachments`：需求附件
- `demand_status_history`：需求状态历史
- `quotes`：报价金额、工期、方案、有效期和状态
- `quote_attachments`：报价附件

一个开发者对同一需求最多保留一条有效报价。报价被选择后生成订单，原需求停止接受新报价。

### 6.3 订单与履约

- `orders`：客户、开发者、需求、报价、金额、佣金率和订单状态
- `order_messages`：订单内留言
- `order_attachments`：留言和履约附件
- `deliveries`：唯一正式交付记录及其版本
- `acceptances`：验收结果和说明
- `order_status_history`：订单状态历史
- `supplement_orders`：补充订单与原订单的关联

首版每个订单只能有一条生效的正式交付记录。开发者在客户验收前可以替换交付版本，但历史版本必须保留。

### 6.4 支付与结算

- `payments`：支付单、金额、渠道状态和渠道单号
- `payment_events`：原始回调摘要与处理结果
- `refunds`：退款申请、审核和渠道状态
- `profit_shares`：开发者分账金额、平台佣金和渠道状态
- `reconciliation_runs`：对账批次和差异结果

金额以最小货币单位整数保存。佣金率在订单创建时固化，后续费率调整不影响已有订单。

### 6.5 评价、争议与治理

- `reviews`：订单完成后的双方评价
- `disputes`：争议原因、诉求、状态和裁决
- `dispute_evidence`：争议证据
- `expert_reviews`：第三方专家评审记录
- `notifications`：站内通知
- `risk_events`：风险事件
- `audit_logs`：管理员和关键系统操作日志

## 7. 订单与支付状态

### 7.1 正常订单状态

`pending_payment → in_progress → delivered → accepted → sharing → completed`

- `pending_payment`：客户已选择报价，等待全额付款
- `in_progress`：支付成功，开发者履约中
- `delivered`：开发者提交正式交付
- `accepted`：客户验收通过，等待分账
- `sharing`：微信分账处理中
- `completed`：分账成功，订单完成

### 7.2 异常状态

- `closed`：未付款超时或双方在付款前取消
- `refund_review`：退款申请待平台处理
- `refunding`：微信退款处理中
- `refunded`：退款完成
- `disputed`：订单进入争议处理
- `share_failed`：分账失败，等待后台重试或人工处理

状态只能通过明确的服务端动作转换。前端不得直接写入订单状态。

### 7.3 交付与验收规则

- 开发者只能在 `in_progress` 状态提交正式交付。
- 正式交付包含交付说明和至少一个附件或交付链接。
- 客户可以验收通过，或拒绝验收并填写具体理由。
- 拒绝验收后订单回到履约处理，但保留全部交付版本和拒绝理由。
- 双方无法协商时，任一方可以申请仲裁。
- 首版不自动按时间验收；未来启用超时自动验收前，必须更新交易协议并增加充分提醒。

### 7.4 退款和仲裁

- 付款后退款必须关联订单和明确原因。
- 未开始履约、开发者无法履约、双方协商解除和仲裁裁决可以触发全额退款。
- 首版不支持部分退款，避免金额、佣金和分账状态复杂化。
- 已完成分账的订单退款需按微信支付能力和签约规则单独处理；上线前必须完成对应测试。
- 仲裁证据包括需求版本、报价、订单留言、附件、交付物、验收记录和状态历史。
- 专家评审只在平台认为有必要时启动，并记录专家身份、意见和利益冲突声明。

## 8. 收费规则

- 平台仅向开发者收成交佣金。
- 建议默认佣金率为订单实付金额的 10%。
- 首批开发者优惠费率由后台配置。
- 退款金额不计佣金。
- 支付通道费首版从平台佣金中承担，不单独向客户展示。
- 分账金额和平台佣金由服务端基于订单固化费率计算。
- 所有费率、计算结果和渠道请求必须可追溯。

## 9. 通知

站内通知为必备能力。以下事件发送站内通知，并可按配置补充短信或邮件：

- 需求审核结果
- 收到新报价
- 报价被选择或未被选择
- 支付成功或异常
- 新订单留言
- 正式交付
- 验收结果
- 退款状态变化
- 仲裁状态变化
- 分账成功或失败

通知发送失败不得回滚业务事务，但必须记录失败并允许重试。

## 10. 安全、风控与审计

- 所有金额由服务端根据数据库中的有效报价计算。
- 支付创建、回调、退款和分账操作使用幂等键。
- 微信支付回调必须验签并防止重复处理。
- 关键状态转换在数据库事务中完成。
- 管理员高风险操作记录操作者、时间、原因、变更前后数据和请求来源。
- 登录、验证码、报价、留言、上传、支付和仲裁接口实施限流。
- 私有文件禁止使用永久公开链接。
- 开发者公开资料和需求必须审核后展示。
- 风控后台至少支持封禁用户、暂停需求、冻结订单操作和标记异常支付。

## 11. 测试策略

### 11.1 单元测试

- 佣金和分账金额计算
- 订单状态转换
- 报价有效性
- 权限与角色判断
- 文件访问权限
- 退款资格判断

### 11.2 集成测试

- 注册、角色切换和开发者认证
- 需求发布与审核
- 报价、选标和订单生成
- 模拟支付到履约
- 正式交付和验收
- 退款与仲裁
- 支付回调、退款回调和分账结果处理
- 对账差异生成

### 11.3 端到端测试

- 客户完整流程
- 开发者完整流程
- 管理员审核、退款、仲裁和异常分账处理流程
- 手机与桌面响应式流程

### 11.4 支付上线验收

- 支付成功、取消、关闭和超时
- 回调延迟、重复、乱序和验签失败
- 支付成功但业务处理失败后的主动查单恢复
- 退款成功、失败和处理中
- 分账成功、失败和重试
- 日账单与平台支付数据对账
- 生产金额、商户号、二级商户和分账接收方校验

## 12. 上线标准

- 客户、开发者和管理员三条主流程均通过端到端测试。
- 不存在已知高危越权或私有文件泄露问题。
- 每笔订单、支付、退款、分账和状态变化均可追踪。
- 支付回调可重复接收且不会重复记账。
- 对账差异可以在后台查看和处理。
- 服务协议、隐私政策、交易规则、退款规则和争议规则已发布。
- 运营主体、联系方式、投诉渠道和数据删除渠道已公开。
- 未取得对应支付能力前，不展示“资金托管”“担保交易”等承诺。

## 13. 首版明确不做

- 定金尾款
- 里程碑分期付款
- 部分退款
- 实时聊天
- 音视频通话
- 用户钱包、充值和余额提现
- 自动匹配或智能推荐算法
- 原生 App 或微信小程序
- 多支付渠道同时首发
- 自动超时验收

## 14. 成功指标

首版上线后前 30 天重点跟踪：

- 通过认证的真实开发者数量
- 审核通过的真实需求数量
- 每条需求收到的有效报价数量
- 选标转化率
- 支付成功率
- 交付和验收完成率
- 退款与争议率
- 平均需求审核时间
- 平均首次报价时间
- 平均订单完成周期

第一阶段目标不是追求大规模流量，而是验证真实需求、真实开发者和真实付款能否稳定完成闭环。
