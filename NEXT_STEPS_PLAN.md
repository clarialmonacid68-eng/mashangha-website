## 一、现状说明（从这里出发）

项目当前是「**第一阶段：本地可验证的交易闭环雏形**」。从认证、需求、报价、选标、模拟付款、交付、验收、评价到争议退款裁决的主流程，已经能在本地跑通并通过测试；运营后台、站内通知、安全基础也已具备初版。

但要成为「真正能上线运营、能收真钱」的平台，还差三类东西：

1. **第一阶段收尾**：把已搭好的骨架补成「自然可用」——主要是 UI 真实可点、运营后台能执行高风险动作、文件能真实上传、监控能看见问题。
2. **上线合规前置**：ICP 备案、生产部署锁定、短信/邮件服务商落地、信任与合规页面补全。这些大多是**外部流程**，不是写代码能解决的，需要尽早启动因为审核周期长。
3. **第二阶段真实资金闭环**：微信支付准入、二级商户进件、真实支付/退款/分账/对账。这一阶段在拿到微信支付**书面准入确认前保持 blocked**。

**贯穿始终的红线**：真实支付上线前，页面文案、报告、销售话术一律使用「模拟支付 / 第一阶段验证中」口径，**不得**宣传资金托管、担保交易、保证退款、自动分账、虚构成交数据。

## 二、工作主线与优先级

下面按「应该先做什么」排序。P0 是上线前的硬阻塞，P1 是第一阶段质量收尾，P2 是真实资金闭环（依赖外部准入）。

### P0 — 上线合规前置（尽早并行启动，审核周期长）

这些不写代码也要先推进，否则代码再完美也无法在中国大陆合规上线：

1. **ICP 备案**：确定生产服务器/接入服务商，提交备案，拿到备案号并在页脚展示。（参考 `docs/feasibility/china-production-readiness.md`，当前 blocked）
2. **生产部署方案锁定**：完成中国大陆移动/联通/电信三网 24 小时网络实测，对比 Vercel+Supabase 与境内方案，锁定生产架构。（参考 `docs/architecture/production-hosting-decision.md`）
3. **短信服务商落地**：腾讯云/阿里云国内短信完成账号实名、签名报备、模板审核（通常 7–10 工作日起）。
4. **邮箱 OTP 到达率**：配置生产 SMTP，对 QQ/163/企业邮箱实测到达率。
5. **信任与合规页面补全**（在正式收集联系方式前）：运营主体/负责人信息、联系方式、隐私说明、服务规则、投诉与纠纷处理、数据删除渠道。规则页框架已在 `app/(marketing)/rules/` 存在，需填真实主体信息。

> 说明：P0 第 1–4 项需要运营者提供决策与资料（见第四节）。在 P0 未完成前，不建议投放任何付费流量。

### P1 — 第一阶段质量收尾（可立即用代码推进）

来自最新审查报告 `docs/reviews/2026-06-16-marketplace-platform-audit.md` 的 P1 建议，目标是把骨架补成自然可用：

   `客户登录 → 发布需求 → 管理员审核 → 开发者报价 → 客户选标 → 页面创建并确认模拟支付 → 开发者交付 → 客户验收 → 模拟结算 → 评价`。
   配套补齐了开发者报价 UI、客户验收/退回/结算/评价 UI、订单列表页、客户需求列表页，并修复侧边栏死链。
2. **真实文件上传闭环**（未完成）：把附件/作品图片接到 Supabase Storage，生成短时签名链接，验证权限。（`lib/storage/` 已有策略骨架）
3. ✅ **运营后台高风险动作**（大部分已完成，2026-06-17）：后台从「查看型」补成「可执行」——
   - ✅ 订单冻结 / 解冻（`/admin/orders`，服务层拦截冻结订单的所有新动作）
   - ✅ 封禁 / 解封用户（`/admin/risk`，按账号 ID）
   - ✅ 暂停 / 恢复需求（`/admin/demands`，暂停后从公开市场与详情页隐藏）

   - ⬜ 退款执行与失败重试（仍未完成，见下方"剩余 P1"）

   所有已实现动作均写审计日志（`lib/domain/admin/governance.ts` / `lib/domain/disputes/service.ts`）；冻结只阻止新动作、不改历史财务记录。

### P2 — 第二阶段：真实微信支付资金闭环（外部准入解锁后才做）

## 三、建议执行节奏（参考）

下面是把上面优先级铺到时间线上的建议，非硬性承诺；真实上线日期受备案与微信支付审核等外部因素制约。

| 阶段 | 重点 | 产出 |
|---|---|---|
| 第 1–2 周 | 启动全部 P0 外部流程（备案、短信报备、SMTP）；同步做 P1-1 UI E2E 与 P1-5 文档更新 | 备案/短信进入审核队列；一条完整 UI 链路绿灯；进度报告刷新 |
| 第 3–4 周 | P1-2 文件上传闭环 + P1-3 运营后台高风险动作（订单冻结、用户封禁、争议裁决 UI、退款执行） | 后台可执行关键运营动作并留审计 |
| 第 5 周 | P1-4 可观测性；补全 P0-5 信任合规页面真实信息；三网网络实测出结果 | 监控可见、合规页就绪、部署方案锁定 |
| 第 6 周 | 第一阶段上线验收（功能 + 合规 + 安全 + 性能），决定是否开放试运营获客 | 第一阶段上线清单全绿 |
| 微信准入后 | 进入 P2 Task 15–19，逐项替换为真实资金闭环 | 真实支付/退款/分账/对账上线 |

如微信支付审核提前完成，可把 P2 前移；若延期，第一阶段只开放撮合与订单管理，继续用线下合同 + 人工收款，不展示资金托管承诺。

## 四、需要运营者/负责人确认的事项（代码无法替你决定）

以下问题不解决，P0 无法推进。建议尽快给出明确答复：

1. **生产部署落点**：用境外（Vercel/Supabase）做预览 + 境内服务器做正式？还是全境内？这决定 ICP 备案路径。
2. **ICP 备案主体**：用哪个公司/个人主体备案？是否已注册运营公司？
3. **线索/申请收件渠道**：第一阶段需求线索和开发者申请发到哪里（企业微信 / 企业邮箱 / 飞书多维表 / 数据库）？
4. **对外展示的联系人信息**：页脚和合规页要公开的负责人 / 微信 / 邮箱。
5. **已有资源盘点**：是否已有 Supabase 生产项目、企业邮箱、企业微信、国内短信账号？
6. **首批真实供给与需求**（参考 `OPERATIONS.md` 的 30 天节奏）：招募 5–10 位真实开发者、获取首批 10 条真实需求，**删除所有示例人物数据，换成真实可核验数据**。
7. **微信支付准入**：何时启动商户申请？由谁负责法务/财务材料？

## 五、风险与红线提醒

- **合规红线**：真实支付商户、退款、分账、对账、生产证书全部验收前，页面与话术只能说「本地模拟全额付款 / 第一阶段交易流程验证中」，对外成交建议继续走线下合同 + 人工收款。
- **数据红线**：上线前必须用真实、本人授权的开发者与案例数据替换全部示例数据，否则涉嫌虚假宣传。
- **测试红线**：涉及数据库的改动，提交前必须导出本地 Supabase env 跑全量 `pnpm vitest run` + `pnpm test:e2e`，仅 `pnpm verify` 会跳过集成测试，不足以代表业务验证。
- **证据保全**：订单、支付、退款、争议、审计日志在法务/财务留存期内不得硬删除（详见 `OPERATIONS.md`）。
- **外部依赖不可控**：备案、短信签名报备、微信支付准入都有不确定的审核等待期，应尽早并行启动，不要排成串行依赖卡住上线。

## 六、给接手 AI 的「下一步」建议

如果你现在就要动手，**风险最低、收益最快**的起点是 P1 中纯代码、不依赖外部准入的项：

1. 先做 **P1-1 完整 UI 点击式 E2E**（验证主流程真实可用，也是后续重构的安全网）。
2. 再做 **P1-5 更新进度报告**（让团队/协作者看到准确现状）。
3. 然后做 **P1-2 文件上传闭环** 与 **P1-3 运营后台高风险动作**。

按项目既有习惯推进：先写失败测试 → 实现 → 跑专项测试 → `pnpm verify`，并同步更新对应 `docs/` 文档。
